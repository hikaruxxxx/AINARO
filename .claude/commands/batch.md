あなたはAINAROの大量生成パイプラインエージェントです。
月5,000〜10,000話規模の生成を管理します。

## 引数

$ARGUMENTS を解析してください:
- 形式A（新規作品群を一括シード＋生成）: `new {数} "{ジャンル}"`
- 形式B（既存作品群の一括追加生成）: `existing {話数/作品}`
- 形式C（混合）: `mix {新規数} {既存話数/作品}`
- 例: `new 5 "悪役令嬢"` → 悪役令嬢ジャンルで5作品を新規作成＋各1話生成
- 例: `existing 10` → 全既存作品にそれぞれ10話追加
- 例: `mix 3 5` → 3作品新規作成 + 既存全作品に5話追加

## 全体フロー

```
計画策定 → [新規作品シード] → [プロット生成] → 本文生成 → 自動QA → 台帳更新 → サマリー
```

---

## Step 1: 計画策定

### 規模の計算
```
新規作品数: {N}
既存作品数: {M}（content/works/ をスキャン）
生成予定話数: {新規×1 + 既存×話数/作品}
推定文字数: {話数 × 4,000字}
```

### ジャンル多様性チェック（新規作品の場合）

既存作品のジャンルを確認し、以下を自動判定:
- 被りが多すぎるジャンルは警告
- サブジャンルの自動差別化（同じ「悪役令嬢」でも設定・主人公・切り口を変える）

### ログライン自動生成（新規作品の場合）

指定ジャンル内でバリエーションを自動生成:

**差別化の軸:**
1. **主人公の属性**: 年齢・職業・性格・能力
2. **状況設定**: どんな問題に直面するか
3. **世界観**: 舞台の特異性
4. **構造**: 連続型 / オムニバス / ループもの / 群像劇
5. **トーン**: シリアス / コメディ / ダーク / ほのぼの

例: 「悪役令嬢」ジャンルで5作品
```
1. "処刑された悪役令嬢が10歳に巻き戻り、今度は聖女ルートを目指す" （ループ×聖女）
2. "悪役令嬢だが領地経営が得意すぎて国が潰れる" （内政チート×コメディ）
3. "悪役令嬢の母親に転生。娘の断罪を防ぐために宮廷工作" （親世代視点×策略）
4. "6人の悪役令嬢が同じ学園に転生してしまった" （群像劇×コメディ）
5. "悪役令嬢が断罪を受け入れて追放された先が、実は楽園だった" （追放×スローライフ）
```

**重要**: 各ログラインを生成したら、既存作品との設定被りをチェックする。

### 計画の表示

```
=== AINARO batch 計画 ===

新規作品:
  1. {slug}: "{ログライン}" ({ジャンル})
  2. {slug}: "{ログライン}" ({ジャンル})
  ...

既存作品:
  {slug1}: ep{N}→ep{N+10}（10話追加）
  {slug2}: ep{N}→ep{N+10}（10話追加）
  ...

合計: {作品数}作品 × {話数}話 = {合計話数}話（約{文字数}万字）
```

## Step 2: 新規作品のシード展開

新規作品がある場合、各作品に対して /seed コマンドを実行:

1. 各作品のログラインからディレクトリ構造と設定ファイルを自動生成
2. 第1話のプロットまで生成

**slug の自動生成ルール:**
- ログラインからキーワードを抽出
- 英語のケバブケースに変換
- 既存slugと被らないことを確認
- 例: "処刑された悪役令嬢が10歳に巻き戻り" → `villainess-rewind-10`

---

## Step 3: プロット一括生成

生成対象の全話について、プロットファイルの有無を確認。
ないものは一括でプロットを自動生成する。

**一括生成時のルール:**
- 同一作品内のプロットは連続して生成（前話のプロットが次話に影響するため）
- テンションカーブとの整合性を事前チェック
- 伏線の設置/回収バランスを5話単位で確認

---

## Step 4: 本文一括生成

各作品・各話を順次生成。/generate コマンドのフローに従う。

**排他制御**: 各作品の生成開始前に generation_locks でロック取得。取得失敗=別プロセス生成中なのでスキップ。全話完了後に解放。30分自動期限切れ。DB未接続時は .generation.lock ファイルで代替。

**生成順序:**
```
作品A: [lock] ep1 → ep2 → ... → epN [unlock]
作品B: [lock] ep1 → ep2 → ... → epN [unlock]
```

（同一作品内は順次。前話の文脈が必要なため）

**進捗表示:**
```
[進捗] {完了数}/{合計数} ({完了%}%)
[{slug}] ep{num} 完了: {文字数}字 | {テンション}
```

---

## Step 5: 自動QA

全話に対して自動校正を実行（/daily の品質ゲートと同じ）:

- S/A/B → 自動通過
- C → 自動修正を試みる
- D → 再生成（最大2回）

**LLM品質スコアリング（第1話のみ）:**

新規作品の第1話に対して、6軸のLLM品質評価を実施する。
以下の基準で1-10の整数を採点し、人気評価APIに渡す:

| 軸 | 評価基準 |
|---|---|
| hook | 冒頭の衝撃度。最初の5行で「おっ」と思わせるか |
| character | キャラの声の独自性。台詞を入れ替えても気づかないなら低い |
| originality | 設定・展開の独自性。「どこかで読んだ」なら3以下 |
| prose | 文章力。リズム、比喩、描写の密度 |
| tension | 緊張感の持続。ずっと先が気になったか |
| pull | 読み終えた瞬間、第2話を開きたいか |

採点基準: 1-2=苦痛、3-4=平凡テンプレ、5-6=及第点、7-8=商業級、9-10=傑出
**目安: 平均は4-5。7以上は全体の15%程度。厳しく評価すること。**

LLMスコアは `analyzePopularity(text, genre, llmScores)` に渡され、
PV予測モデル（v3, Spearman=0.733）の精度向上に使われる。

**品質ゲート追加基準:**
- LLM total（6軸平均）≧ 5.0 → 自動通過の補助条件
- LLM total < 3.0 → D判定と同等（再生成対象）

**大量生成時の追加チェック:**
- **作品間の表現被り**: 異なる作品で同じ比喩・展開が出ていないか
- **キャラ名の被り**: 異なる作品で似た名前のキャラがいないか
- **冒頭パターンの多様性**: 「目を覚ますと〜」が何作品にも出ていないか

---

## Step 6: 台帳一括更新

全作品の以下を更新（DB優先、フォールバック: Markdownファイル）:
- **伏線台帳**: `foreshadowing_items` テーブル（新規INSERT + 回収済UPDATE）
- **読者既知情報**: `reader_knowledge_items` テーブル（各話の新規情報をINSERT）
- **ドラマティック・アイロニー**: `dramatic_irony_items` テーブル（状態更新）
- **ワールドステート**: `_world_state/ep{num}_snapshot.md`（適応的間隔: ep1-50は5話ごと、ep51以降は10話ごと）

DB未接続時は従来のMarkdownファイルに書き込む。

## Step 7: サマリー

```
=== AINARO batch 完了 ===

新規作品: {N}作品作成
既存作品: {M}作品に追加生成

生成結果:
  合計: {話数}話 / {文字数}万字
  品質分布: S{x} A{y} B{z} C{w} D{v}
  自動通過率: {%}%
  例外レビュー対象: {N}話

作品別サマリー:
  {slug1}: {N}話 | 平均{グレード}({スコア}) | 公開可能{N}話
  {slug2}: {N}話 | 平均{グレード}({スコア}) | 公開可能{N}話
  ...

例外レビュー対象:
  {slug} ep{N}: {問題の概要}
  ...

次のステップ:
  /daily {slug} で個別レビュー
  例外レビュー対象は手動確認が必要
```

## 重要なルール

- **多様性を最優先**: 同じようなキャラ・設定・展開の作品を量産しない
- **品質のフロア**: D判定が全体の10%を超えたら一旦停止して原因を調査
- **ログライン品質**: 「面白そう」と思えないログラインは採用しない。数より質
- **slug管理**: 一意で、内容を推測できるslugにする
- **状態ファイルの整合性**: 大量生成後に台帳の整合性が崩れていないか確認
- **段階的スケール**: いきなり100作品×100話ではなく、5作品×10話から始めて品質を確認
