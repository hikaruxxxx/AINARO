## 7. フィードバックループ

### 7.1 ループの流れ

```
生成 → 校正(自動) → レビュー(人間) → 配信 → 計測(自動)
  ▲                                              │
  └──────────── ナレッジ更新 ◄────────────────────┘
```

### 7.2 学習の3段階

**即時学習（同日）**: 校正・レビューの修正内容
- /review で記録した修正理由が、次の /generate に反映
- 「テンポが遅い」→ ��回生成で短文率・テンポパラメータを調整

**短期学習（週次）**: 読者行動データ
- 読了率が低い話 → 何が悪かったかを特徴量で分析
- 読了率が高い話 → 何が効いたかを特徴量で分析
- data/feedback/reader_signals/ を集計してベンチマーク更新

**長期学習（月次）**: ベンチマーク再キャリブレーション
- クロールデータの追加（新作の人気作品）
- 自社作品の蓄積データによるベンチマーク補正
- 人気予測モデルの重み調整

### 7.3 自社作品データの特別扱い

自社作品は**すべての指標が揃う**唯一のデータソース:
- テキスト特徴量（生成時に自動計算）
- 校正スコア（校正時に自動保存）
- 人間レビュー（/review で記録）
- 読者行動（配信後に自動収集）

こ���により「テキスト特徴量 → 読者行動」の直接的な因果関係を学習できる。
クロールデータ（なろう等）は「テキスト → ポイント」の粗い相関しか取れないが、
自社デ��タは「テキスト → 読了率・次話遷移率」の精密な相関が取れる。

## 11. 仮説自動発見メカニズム

### 問題意識

現在のエージェントは「疑問率が高い→人気」等の**固定仮説**で動いている。
しかし本当に重要な特徴量は、我々がまだ気づいていないものかもしれない。
「見逃している仮説を自動で発見し、検証し、取り込む」仕組みが必要。

### 仮説の3階層

```
┌────────────────────────────────────────────┐
│ L1: ジャンル横断仮説（全作品共通）           │
│   「疑問率が高い話は次話遷移率が高い」       │
│   「感情語を直接書くほど読了率が下がる」     │
├────────────────────────────────────────────┤
│ L2: ジャンル固有仮説                        │
│   「悪役令嬢ものは内面独白率30%以上で人気↑」 │
│   「異世界恋愛は会話率40%+が最適」           │
│   「ホラーは短文率が高いほど読了率↑」        │
├────────────────────────────────────────────┤
│ L3: 作品固有仮説                            │
│   「この作品はツッコミ独白が多い話ほど好評」  │
│   「クロードが登場する話は次話遷移率が高い」  │
└────────────────────────────────────────────┘
```

### 未検証の仮説候補（初期リスト）

現在の特徴量設計が見落としている可能性のある軸:

| カテゴリ | 仮説 | 検証方法 |
|---|---|---|
| **話数位置** | 第1話と第10話では成功パターンが違う（1話はフック重視、中盤はキャラ深堀） | ep番号別に特徴量と人気の相関を分析 |
| **ジャンル内分類** | 悪役令嬢内でも「復讐型」「コメディ型」「シリアス型」で最適パラメータが違う | クラスタリングで自動分類→tier相関 |
| **情報密度** | 固有名詞・設定説明の密度が高すぎると離脱する | 固有名詞率を特徴量に追加 |
| **キャラ登場数** | 1話に登場するキャラ数が多すぎると混乱→離脱 | 「」直前のキャラ名をカウント |
| **感情の方向性** | ポジティブ→ネガティブの落差（感情反転）が大きい話は引きが強い | 前半/後半の感情極性を比較 |
| **テンポの変化** | 話の中でテンポが一定 vs 緩急がある | 文長の時系列変化パターン |
| **冒頭の型** | 会話開始 vs 描写開始 vs 独白開始で読了率が違う | 冒頭パターンを分類して相関 |
| **引きの型** | 疑問型 vs 危機型 vs 感情型 vs 新情報型で次話遷移率が違う | 末尾パターンを分類して相関 |
| **シリーズ構造** | 章の長さ、クライマックスの配置が完走率に影響 | _meta.jsonの章構成を分析 |

### スキル: /hypothesize

**仮説の自動生成・検証サイクル**。これがFBループの「知能」にあたる。

```
/hypothesize
```

**処理フロー**:
1. data/benchmarks/ と data/feedback/ の最新データを読む
2. 現在の仮説台帳（data/hypotheses/ledger.json）を読む
3. LLM（Claude）がデータを見て新しい仮説を提案
4. 検証可能な仮説に対して統計テストを自動実行
5. 有意な結果が出た仮説を「検証済み」に昇格
6. 検証済み仮説をエージェントの重み・閾値に反映

**仮説台帳の構造**:
```json
{
  "hypotheses": [
    {
      "id": "H001",
      "statement": "異世界恋愛ジャンルでは会話率35-45%の作品がtop tierに多い",
      "status": "validated",
      "evidence": { "spearman": 0.62, "pValue": 0.003, "n": 40 },
      "action": "popularity-evaluationのdialogueRatio重みをジャンル別に調整",
      "createdAt": "2026-04-10",
      "validatedAt": "2026-04-12"
    },
    {
      "id": "H002",
      "statement": "第1話の冒頭3文が20字以下の短文で始まる作品はbookmark率が高い",
      "status": "testing",
      "evidence": null,
      "createdAt": "2026-04-13"
    }
  ]
}
```

### ジャンル細分化の判断基準

「ジャンルごとに分けるべきか」自体を仮説として扱う:

1. まず全ジャンル一括のベンチマークで人気予測を実行
2. ジャンル別ベンチマークで同じ予測を実行
3. 予測精度（相関係数）が有意に改善したジャンルのみ分離
4. 改善しないジャンルは統合のまま

→ **データが答えを出す**。事前に人間が判断しない。

## 12. 運営の簡素化

### 問題意識

セクション9の運営フローは9つのコマンドを手動実行する設計。
1人運営で毎日これをやるのは現実的でない。

### 設計原則: 人間の判断は「面白いか」の1点だけ

人間がやるべきこと:
- 生成されたテキストを**読んで**、OK/NGを**判断**する

人間がやるべきでないこと:
- コマンドの実行
- パラメータの指定
- ファイルの管理
- スケジューリング

### 統合スキル: /daily

```
/daily test-villainess
```

**これ1つで1日の作業が完了する**。

**処理フロー**:
```
1. 状態確認
   ├ 前回どこまで生成したか（ep最新番号を取得）
   ├ 未レビューのエピソードがあるか
   └ 本日の公開予定があるか

2. 生成（必要なら）
   ├ _plot.md から次に書くべき話を特定
   ├ 3-4話を順次生成
   └ 各話を自動で校正（/proofread相当）

3. レビュー待ちリストを表示
   ├ 「ep005: 校正A(82点), 人気予測B(68), AI検出42」
   ├ 「ep006: 校正B(65点), 人気予測C(52), AI検出38」
   └ 「ep007: 校正C(45点) ← 要注意」

4. 人間レビュー（対話的）
   ├ 各話のテキストを表示
   ├ 「OK / NG / 修正して再生成」を聞く
   ├ NGの場合は理由を聞いてFB記録
   └ 修正の場合はその場で再生成→再校正

5. 公開キュー
   ├ OKのエピソードを公開スケジュールに追加
   ├ 「12:00 ep005 / 18:00 ep006 / 21:00 ep007 でOK？」
   └ 確認後に自動スケジュール
```

## まとめ

- **トークン消費は気にしない**（Maxプラン定額）。コンテキストを贅沢に使う
- **ナレッジ層がすべての鍵**。ベンチマーク + FB履歴の質が生成品質を決める
- **人間の判断は「面白いか」の1点**に集中。残りはエージェントが担当
- **自社データが最強の学習信号**。クロールデータは初期ブートストラップ用
- **仮説はデータが検証する**。ジャンル分割の要否もシステムが判断する
- **日常運用は `/daily` 1コマンド**。45分で1日分の作業が完了する
