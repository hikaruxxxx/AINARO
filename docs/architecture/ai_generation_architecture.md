# AI小説生成アーキテクチャ設計書

## 概要

Novelisの小説制作パイプラインにおけるAIツール構成と生成フローの設計。
1人運営・月100話・女性向けファンタジーの要件に最適化。

---

## 1. ツール構成

### 設計思想

**「設定ファイルの質 = 生成品質」**

ツール選択よりも、AIが参照する設定（キャラ・世界観・文体・テンポ）の構造と蓄積が
出力品質を決定する。外部サービス（NovelCrafter, World Anvil, Campfire等）は
いずれも自動化パイプラインに使えるAPIを持たないため、自前管理が必須。

### 段階的アプローチ

```
Phase 0 前半（〜1ヶ月目）: Markdownファイル管理 + Claude Max
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ・最速で生成開始し、品質を検証
  ・設定の「型」を試行錯誤で固める
  ・Claude Codeがファイル直接読み書き → 最もスムーズ

Phase 0 後半（2-3ヶ月目）: Supabase DB に移行
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ・Markdownで固まった構造をDBスキーマ化
  ・複数作品並行 + 読者データ連携
  ・生成パイプラインのスクリプト化・自動化
```

### 採用ツールスタック

```
┌──────────────────────────────────────────────────────┐
│  Phase 0 前半                                         │
│                                                      │
│  [設定層]           Markdownファイル（content/）        │
│    キャラ設定 / 世界観 / 文体プロファイル / プロット      │
│    → Claude Codeが直接読み書き                         │
│    ↓                                                 │
│  [生成・校正]       Claude Opus 4.6（Maxプラン $200/月）│
│    本文生成 → 校正 → ファイル保存                       │
│    ↓                                                 │
│  [最終確認]         人間（自分）                         │
│    品質判定 → Supabase投稿                             │
├──────────────────────────────────────────────────────┤
│  Phase 0 後半（移行後）                                 │
│                                                      │
│  [設定DB層]         Supabase（自社DB）                  │
│    Markdownで検証済みの構造をDB化                       │
│    配信DB・読者データと同一基盤                          │
│    ↓                                                 │
│  [生成・校正]       Claude Opus 4.6（Maxプラン）         │
│    DB → コンテキスト自動組立 → 生成 → 校正              │
│    ↓                                                 │
│  [最終確認]         人間（自分）                         │
│    品質判定 → 自動投稿                                  │
└──────────────────────────────────────────────────────┘
```

### 外部ツールを採用しない理由

| ツール | 不採用理由 |
|---|---|
| **NovelCrafter** | Codexのプログラム的API**なし**（手動zip出力のみ）。自動化パイプラインに組み込めない |
| **World Anvil** | APIはあるがGrandmaster以上限定。小説生成パイプラインとの統合は想定外の設計 |
| **Campfire Write** | API**なし**。モジュール型で柔軟だが手動UI前提 |
| **Sudowrite** | クレジット制で量産に不向き。Opusの散文品質で十分代替可能 |
| **NovelAI** | 日本語散文品質がOpusに劣る |
| **Nola / ストーリープロッター** | 日本製だがAPI**なし**。個人利用UIのみ |

> いずれも「個人作家が1作品を手動で管理するUI」であり、
> 「月100話を自動生成するパイプラインのデータソース」には使えない。

## 2. コンテンツディレクトリ構成（Phase 0 前半）

```
content/
├── references/                     # 参考資料・分析結果
│   ├── genre_villainess.md          # 悪役令嬢系の構造分析
│   └── tempo_patterns.md            # テンポパターン集
├── style/                          # 共通スタイル設定
│   ├── base_guidelines.md           # 共通文体ガイドライン
│   └── blacklist.md                 # AI NG表現リスト
└── works/                          # 作品ごとのディレクトリ
    └── {作品slug}/
        ├── _settings.md             # キャラ・世界観設定
        ├── _style.md                # 文体プロファイル（パラメータ+例文）
        ├── _plot.md                 # 全体プロット + 各話プロット
        ├── ep001.md                 # 生成済みエピソード
        ├── ep002.md
        └── ...
```

### Claude Codeとの相性

- Claude Codeはファイル読み書きが最も得意な操作
- `_settings.md` を読み込み → 本文生成 → `ep001.md` に書き出し、が1つの会話内で完結
- 設定変更もファイル編集で即反映。DB接続・マイグレーション不要
- Maxプランなのでリトライ・試行錯誤にコスト制約なし

## 4. 生成パイプライン

### Phase 0 前半：Claude Code直接実行

```
1. 設定ファイルを読み込む
   Claude Codeが content/works/{slug}/ 配下の設定ファイルを読む
   ├ _settings.md → キャラ・世界観
   ├ _style.md    → 文体パラメータ
   ├ _plot.md     → 当話プロット
   └ content/style/blacklist.md → NG表現

2. 本文生成
   Opus 4.6が設定を参照して4,000字の本文を生成

3. 校正（同一会話内）
   生成した本文を設定と突合し、文体・整合性を校正

4. ファイル保存
   ep{number}.md として保存

5. 人間レビュー（15-20分）

6. Supabaseに投稿（手動 or スクリプト）
```

### 1日の運用フロー

```
08:00  Claude Codeで3-4話を生成（各話10-15分 × 3-4本 = 約1時間）
       └ _settings.md + _plot.md を読み込み → 生成 → 校正 → 保存

09:00  人間レビュー（約1時間）
       ├ OK → 投稿キューに追加
       └ NG → その場でリトライ（Maxプランなのでコスト0）

12:00  第1話 公開
18:00  第2話 公開
21:00  第3話 公開

週末   プロット作成・新作企画・品質改善
```

## 5. プロンプト設計方針

### 文体プロファイル（_style.md の構造）

```yaml
# 文体パラメータ
pov: first_person           # 視点: 一人称
tempo: fast                  # テンポ: slow / medium / fast
sentence_length_avg: 20      # 平均文長（文字数）
dialogue_ratio: 0.4          # 会話文の割合
inner_monologue_ratio: 0.25  # 内面独白の割合
description_density: medium  # 描写密度: sparse / medium / rich
humor_level: high            # ユーモア度
vocabulary_level: casual     # 語彙水準: casual / standard / literary
line_break_frequency: frequent # 改行頻度（スマホ読み意識）
```

### 話ごとのテンポ動的調整

```
_plot.md 内でテンポを話ごとに指定:

## 第1話「覚醒」
テンポ: fast（引き込み重視）
シーン1: ...
シーン2: ...
引き: ...

## 第5話「日常」
テンポ: slow（キャラ掘り下げ、会話多め）
...
```

### 共通文体ガイドライン（base_guidelines.md）

```
【文体ルール】
- 読点は1文に2つまで。テンポを殺さない。
- 体言止め・倒置法でリズムを作る。
- 感情は「説明」ではなく「描写」で伝える。
  ×「悲しかった」→ ○「視界がにじんだ」
- 五感描写を各シーンに最低1つ。

【構成ルール】
- 冒頭3行で引き込む（謎・衝撃・感情）
- 末尾は「引き」で終わる
- 1話4,000字。シーン転換は最大2回。

【禁止】
- 「〜のだった」「〜なのである」の多用（1話2回まで）
- 同じ表現の近接反復（3文以内に同語彙禁止）
- 設定との矛盾
```

## 6. 品質管理

### Opus校正の観点

| チェック観点 | 判定内容 |
|---|---|
| 文体一貫性 | _style.mdのパラメータと実際の出力の乖離 |
| テンポ | _plot.mdで指定されたテンポと文章リズムの一致 |
| 行間の効果 | 改行・空白行が感情の「間」として機能しているか |
| キャラ口調 | _settings.mdの口調設定との一致 |
| 引きの強度 | _plot.mdのcliffhanger意図が末尾で実現されているか |
| 伏線整合性 | foreshadowing / callbacksが本文に織り込まれているか |
| NG表現 | blacklist.mdに該当する表現の検出 |
| 設定矛盾 | _settings.mdとの不一致 |

### 人間レビューチェックリスト

- [ ] 読んで面白いか（最重要）
- [ ] キャラが「そのキャラらしい」か
- [ ] 前話からの流れが自然か
- [ ] 引きが機能しているか
- [ ] 不自然な日本語がないか

## 7. Phase 0後半：Supabase移行計画

Markdownで検証・確定した構造をDBスキーマ化する。

### 移行トリガー（以下のいずれかに該当したら移行）

- 並行作品数が3作品を超えた
- 読者データ（daily_stats）との連携が必要になった
- 手動ファイル管理が煩雑になった

### 移行内容

| Markdownファイル | → Supabaseテーブル |
|---|---|
| `_settings.md`（キャラ部分） | `characters` |
| `_settings.md`（世界観部分） | `worldbuilding` |
| `_style.md` | `style_profiles` |
| `_plot.md`（全体構造） | `plot_templates` |
| `_plot.md`（各話） | `episode_plots` |
| `references/*.md` | `reference_analyses` |
| `blacklist.md` | `blacklisted_expressions` |

> テーブル定義の詳細はMarkdownで型が固まった段階で確定する。
> 先にスキーマを決め打ちしない。

## 8. リスクと対策

| リスク | 対策 |
|---|---|
| 文体の均質化 | 作品ごとに_style.mdを明確に差別化。パラメータの組み合わせで変化 |
| Maxプランの利用制限 | 公式ポリシー確認。制限時はAPI従量課金をバックアップ |
| モデル更新による品質変動 | _style.md内にreference_examples（参考例文）を蓄積し再現性を確保 |
| 著作権リスク | 既存作品の本文入力は禁止。構造パターンの分析のみ |
| Markdown管理の限界 | 3作品超でSupabase移行。移行はパーサーで一括実行 |

---

## まとめ

- **Phase 0前半はMarkdown + Claude Max**で最速スタート。DB構築を待たない
- **コスト**: 月$200定額。リトライ・試行錯誤に制約なし
- **品質の鍵**: 設定ファイル（_settings.md / _style.md）の質。「何を参照させるか」がすべて
- **外部ツールは不採用**: 自動化パイプラインにAPI連携できるサービスは実質ない
- **Supabase移行は実績ベースで判断**: Markdownで型が固まってから
