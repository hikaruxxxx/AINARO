# 自己強化学習ループ設計

## 概要

AINAROプラットフォームを「使われるほどコンテンツが良くなり、良くなるほど使われる」自己強化エコシステムとして設計する。

囲碁AIが自己対局で強くなったように、AINAROは「大量生成→読者評価→パターン抽出→生成改善」のループを回すことで、AI小説の品質を自律的に向上させる。

## ループ全体図

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐      │
│  │ 生成     │───▶│ 配信     │───▶│ 計測     │      │
│  │ Engine   │    │ Platform │    │ Sensor   │      │
│  └──────────┘    └──────────┘    └──────────┘      │
│       ▲                               │            │
│       │                               ▼            │
│  ┌──────────┐                   ┌──────────┐      │
│  │ 適応     │◀──────────────────│ 分析     │      │
│  │ Adapter  │                   │ Analyzer │      │
│  └──────────┘                   └──────────┘      │
│                                                     │
└─────────────────────────────────────────────────────┘
```

各コンポーネントの役割：

| コンポーネント | 役割 | 入力 | 出力 |
|---|---|---|---|
| 生成 Engine | コンテンツ生成 | 生成パラメータ + 作品設定 | エピソード本文 |
| 配信 Platform | 読者への配信・UX | エピソード | 読者行動イベント |
| 計測 Sensor | 読者行動の記録 | reading_events | 構造化メトリクス |
| 分析 Analyzer | パターン発見・仮説生成 | メトリクス + 本文 | インサイト |
| 適応 Adapter | 生成パラメータの調整 | インサイト | 更新された生成指示 |

## 1. 計測 Sensor（データ収集層）

### 既存基盤
- `reading_events` テーブル（start/progress/complete/next/bookmark/drop）
- `daily_stats` テーブル（日次集計）

### 追加が必要なデータポイント

#### 1a. エピソード内行動の粒度向上
```
reading_events への追加フィールド:
- scroll_speed_changes: float[]    -- 読速度の変化点（減速=集中、加速=飛ばし読み）
- pause_positions: float[]         -- 一時停止した位置（scroll_depth）
- reread_ranges: [float, float][]  -- 読み返した範囲
```

**目的**: 「どの段落が読者を引きつけたか」をエピソード内で特定する。読了率だけでは「どこが良かったか」がわからない。

#### 1b. 読者セグメント情報
```
reader_profiles テーブル:
- reader_id: uuid
- preferred_genres: text[]         -- 読了率が高いジャンル（行動から自動推定）
- reading_pace: enum               -- fast/medium/slow
- taste_cluster: int               -- 協調フィルタリングで自動分類
- lifetime_episodes_read: int
- first_seen_at / last_seen_at: timestamp
```

**目的**: 「誰に面白いか」を区別する。全読者平均では情報が潰れる。

#### 1c. 比較実験メタデータ
```
experiments テーブル:
- experiment_id: uuid
- experiment_type: enum            -- 'ab_opening' | 'ab_pacing' | 'ab_style' | 'ab_genre_mix'
- variant_a_episode_id: uuid
- variant_b_episode_id: uuid
- target_metric: text              -- 'completion_rate' | 'next_rate'
- status: enum                     -- 'running' | 'concluded'
- winner: enum                     -- 'a' | 'b' | 'inconclusive'
- sample_size: int
- confidence: float
```

**目的**: 「何が効いたか」を因果的に特定する。観察データだけでは交絡を排除できない。

## 2. 分析 Analyzer（パターン発見層）

### 2a. エピソードレベル分析
各エピソードに対して、読者行動から「品質シグナル」を自動算出。

```
episode_signals テーブル:
- episode_id: uuid
- completion_rate: float           -- 読了率
- next_transition_rate: float      -- 次話遷移率
- avg_reading_time_ratio: float    -- 想定読了時間に対する実時間比（>1=じっくり読まれた）
- drop_cliff_position: float       -- 最大離脱ポイント（0-1）
- engagement_curve: float[]        -- 区間ごとのエンゲージメント推移
- reread_density: float            -- 読み返し頻度
- quality_signal: float            -- 上記の加重合成スコア（0-100）
```

### 2b. パターン抽出（週次バッチ）
「高スコアエピソードに共通するパターンは何か」をLLMで分析。

```
処理フロー:
1. quality_signal 上位20% と 下位20% のエピソードを抽出
2. それぞれの本文をLLMに投入し、構造的特徴を抽出:
   - 冒頭の型（会話開始/状況描写/内面独白/アクション）
   - テンション推移パターン
   - シーン転換の頻度と位置
   - 対話/地の文の比率
   - 感情語の密度と分布
   - フック（引き）の種類と配置
3. 上位・下位の差分から「効くパターン」を仮説として出力
```

出力例:
```json
{
  "finding": "読了率80%超のエピソードは、冒頭500字以内に対話を含む割合が92%",
  "confidence": "high",
  "sample_size": 47,
  "actionable": true,
  "suggested_rule": "冒頭は対話または行動で始める。状況描写からの開始を避ける"
}
```

### 2c. ジャンル×パターンのクロス分析
品質予測分析で判明した「ジャンル選択が最大要因」を活かす。

```
genre_pattern_matrix テーブル:
- genre: text
- pattern_id: text
- avg_quality_signal: float
- sample_size: int
- trend: enum                      -- 'rising' | 'stable' | 'declining'
```

**目的**: 「このジャンルではこのパターンが効く」という条件付き知識を蓄積。

## 3. 適応 Adapter（生成改善層）

分析結果を生成パイプラインに反映する仕組み。

### 3a. 生成ルールの自動更新

```
content/style/ 配下のルールファイルを段階的に更新:

手動層（人間が編集）:
  base_guidelines.md      -- 不変の基本原則
  blacklist.md            -- NGパターン

自動層（Adapterが更新）:
  learned_patterns.md     -- 分析から発見したパターン（週次更新）
  genre_specific/
    {genre}_patterns.md   -- ジャンル別の効くパターン
  anti_patterns.md        -- 離脱を引き起こすパターン（避けるべきもの）
```

**重要な制約**: 自動層の更新は「追記」のみ。既存ルールの削除・変更は人間の承認を必要とする。暴走防止。

### 3b. プロット生成への反映

```
plot-arcエージェントへの追加入力:

1. テンションカーブのテンプレート
   - 高スコア作品のテンションカーブ平均を「参考曲線」として提供
   - 強制ではなく参考（創造性を潰さない）

2. エピソード構成の推奨パラメータ
   - 推奨文字数レンジ（ジャンル別）
   - 対話/地の文比率の推奨値
   - シーン数の推奨値

3. 避けるべきパターンリスト
   - anti_patterns.md から自動注入
```

### 3c. A/Bテスト自動設計

Analyzerの仮説を検証するための実験を自動提案。

```
仮説: 「冒頭を対話で始めると読了率が上がる」
  ↓
自動提案:
  - 次回の新規エピソード3本で、対話開始版と描写開始版を生成
  - 読者をランダムに振り分け
  - 1週間後に読了率を比較
  - 有意差があれば learned_patterns.md に昇格
```

## 4. 生成 Engine（コンテンツ生成層）

### 既存基盤
- batch.md: 大量生成パイプライン
- generate.md: エピソード単位生成
- plot-arc.md: アーク単位プロット生成

### ループ統合のための拡張

#### 4a. 生成時のコンテキスト階層に「学習層」を追加

```
現在のコンテキスト階層:
  L1: 共通ルール（base_guidelines等）
  L2: 作品設定
  L3: 物語状態
  L4: 話固有
  L5: フィードバック（過去5件レビュー）

追加:
  L6: 学習パターン（learned_patterns.md + genre_specific/{genre}_patterns.md）
  L7: 実験指示（A/Bテスト中の場合、バリアント指定）
```

#### 4b. 生成メタデータの記録

```
episode_generation_meta テーブル:
- episode_id: uuid
- generation_timestamp: timestamp
- model_version: text
- prompt_hash: text                -- 生成に使ったプロンプトのハッシュ
- applied_patterns: text[]         -- 適用した学習パターンのID
- experiment_id: uuid | null       -- A/Bテスト参加時
- variant: text | null
- generation_cost_usd: float
```

**目的**: 「どの生成条件が良い結果を出したか」をトレース可能にする。

## 5. 配信 Platform（読者接点層）

### 読者を増やす（ループの起点）

読者がいなければループは回らない。配信層の役割は「読者を連れてきて、離さない」こと。

#### 5a. 集客ファネル
```
認知（SEO/SNS/口コミ）
  ↓
初回訪問（ランディング）
  ↓  ← ここの離脱を最小化するUX
試し読み（1-3話）
  ↓  ← ここの読了率が全体の品質指標
継続読者（ブックマーク/通知登録）
  ↓
データ貢献者（行動データが蓄積）
```

#### 5b. 読者体験の最適化もループに組み込む
```
計測対象:
- ページ読み込み速度と離脱率の相関
- フォントサイズ/行間の変更と読了率の相関
- 次話への導線UIと次話遷移率の相関
- 更新通知のタイミングと再訪率の相関
```

UXの改善も「仮説→計測→改善」ループで回す。

## 6. フライホイール効果（自己強化の構造）

```
  より多くの読者
       │
       ▼
  より多くの行動データ
       │
       ▼
  より精密な「面白さ」の理解
       │
       ▼
  より面白いコンテンツの生成
       │
       ▼
  より高い読了率・口コミ
       │
       ▼
  より多くの読者（最初に戻る）
```

### フライホイールの加速要因

1. **データのネットワーク効果**: 読者が増えるほど、ニッチな嗜好パターンも検出可能になる
2. **パターンの複利効果**: 発見したパターンは全作品に適用可能。1つの発見が全体の底上げになる
3. **実験速度の向上**: 作品数が増えるほど、同時に走れるA/Bテストが増える
4. **コスト逓減**: パターンが蓄積するほど、生成時の試行錯誤（=コスト）が減る

### フライホイールの減速リスクと対策

| リスク | 現象 | 対策 |
|---|---|---|
| 局所最適 | 特定パターンに収束し、多様性が失われる | 全生成の20%を「探索枠」として、学習パターンを適用しない |
| 過学習 | 少数読者の嗜好に過剰適応 | 読者セグメント別に評価。全体最適ではなくセグメント別最適を追う |
| データ汚染 | Bot/異常行動がデータを歪める | reading_eventsにanomaly検知。異常パターンの自動除外 |
| 品質劣化スパイラル | 低品質→読者離脱→データ不足→さらに低品質 | Phase 0では人間レビューをゲートキーパーとして維持 |

## 7. Phase別ロードマップ

### Phase 0（現在〜6ヶ月）: 基盤構築 + 初期データ収集

```
目標: ループを回す最低限の基盤を作る
読者目標: 3,000 UU/月
作品目標: 10-20作品、各10話以上

必須実装:
☐ reading_events の計測が正しく動作することを確認
☐ daily_stats の自動集計バッチ
☐ エピソード別quality_signal算出
☐ 作品別ダッシュボード（読了率・次話遷移率・離脱ポイント）

生成側:
☐ batch パイプラインの安定稼働
☐ 3話パイロット → 読了率ゲート → 連載化 の選別フロー

この段階では:
- 分析は手動（人間がダッシュボードを見て判断）
- パターン抽出も手動（気づいたことをlearned_patterns.mdに書く）
- ループは月次で人間が回す
```

### Phase 1（6〜12ヶ月）: 半自動ループ

```
目標: 分析の自動化、生成への反映を半自動化
読者目標: 10,000 UU/月
作品目標: 50作品以上

追加実装:
☐ パターン抽出の週次バッチ（LLMによる上位/下位エピソード比較分析）
☐ learned_patterns.md の自動更新（人間承認ゲート付き）
☐ A/Bテスト基盤（experiments テーブル + 読者振り分けロジック）
☐ reader_profiles の自動構築
☐ 読者セグメント別の品質評価

この段階では:
- パターン発見は自動、適用は人間が確認して反映
- A/Bテストは月2-3本
- ループは週次で半自動
```

### Phase 2（12〜18ヶ月）: 全自動ループ

```
目標: 人間は例外対応のみ。ループが自走する
読者目標: 30,000+ UU/月
作品目標: 100作品以上

追加実装:
☐ Adapter の自動化（パターン発見→ルール更新→生成反映が全自動）
☐ A/Bテストの自動設計・自動実行・自動判定
☐ 生成メタデータの全記録とトレーサビリティ
☐ 異常検知と自動ロールバック
☐ ジャンル×パターンのクロス分析自動化

この段階では:
- 人間は「ルール削除」と「例外レビュー」のみ
- ループは日次〜リアルタイムで自走
- 新しいジャンル/パターンの探索も自動提案
```

## 8. 成功指標

ループが正しく機能しているかの判定基準:

| 指標 | 測定方法 | 目標 |
|---|---|---|
| 品質トレンド | 月別の平均quality_signal | 単調増加 |
| 学習速度 | 新パターン発見から全作品反映までの日数 | Phase 0: 30日 → Phase 2: 3日 |
| 実験効率 | A/Bテストの有意結果率 | 30%以上 |
| 探索vs活用 | 探索枠から生まれた高スコアパターン数 | 月1件以上 |
| コスト効率 | quality_signal 1pt向上あたりのAPI費用 | 逓減傾向 |

## 9. 他プラットフォームとの構造的差別化

このループが回り始めると、以下の点で既存プラットフォームとの差が開く:

1. **出版社**: 読者行動データをコンテンツ生成に直接フィードバックする仕組みがない。編集者の勘と経験に依存
2. **なろう/カクヨム**: データは持っているが、生成側を制御できない（作家は人間）
3. **他のAI小説サービス**: 生成はできるが、読者データを持っていない

AINAROだけが「生成と配信の両方を制御できる」ため、ループを閉じることができる。これが最大の構造的優位性。
