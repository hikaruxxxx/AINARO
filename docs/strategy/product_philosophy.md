# Novelis プロダクト哲学

**最上位原則: 読者にとっての面白さを最大化する**

---

## 1. なぜこの哲学か

Web小説プラットフォームの既存プレイヤーは、暗黙のうちに「作品数の最大化」を価値基準にしている。誰でも投稿でき、作品は無審査で公開され、ランキングはPVやブックマーク数で決まる。結果として読者は、大量の作品から面白いものを自力で探すコストを負っている。

Novelisはこの前提を覆す。**読者が「ここに来れば面白い作品に出会える」と信頼できる場所**を作る。すべてのプロダクト判断は、この信頼の構築に向かっているかどうかで評価する。

---

## 2. 判断基準

あらゆる意思決定の場面で、一つの問いを立てる。

> **「これは読者にとって、より面白い体験になるか？」**

この問いにYesと答えられない機能・施策・変更は、実装しない。

---

## 3. 原則

### 3.1 面白さは測定する

感覚ではなくデータで判断する。面白さの代理指標として以下を用いる。

| 指標 | 意味 | 優先度 |
|---|---|---|
| **読了率** | エピソードを最後まで読んだ割合 | 最高 |
| **完読率** | 作品の最新話まで追いついた読者の割合 | 高 |
| **再訪率** | 翌日以降にサイトに戻ってきた割合 | 高 |
| **継続読了率** | 次のエピソードに進んだ割合 | 高 |
| **離脱ポイント** | 読者が読むのをやめた箇所 | 改善用 |

PV数・ブックマーク数・フォロワー数は参考値であり、面白さの指標ではない。ランキングやレコメンドの主軸には使わない。

### 3.2 面白さは改善し続ける

AIを活用した制作だからこそ、データに基づく改善ループを高速に回せる。

```
作品公開
  ↓
読者行動データ収集（読了率・離脱ポイント）
  ↓
改善示唆の導出（「第3話中盤で40%離脱 → テンポの問題」）
  ↓
次話・次作の制作に反映
  ↓
読了率の変化を検証
```

このループを自社制作（Phase 0-1）で回し、実証した上で外部作家にもデータダッシュボードとして開放する（Phase 2）。

### 3.3 面白いものには対価が伴う

面白さは無料である必要はない。面白いものにお金を払うのは正当であり、対価が循環することでエコシステム全体がより面白いものを生み出せる。

- **課金の判断基準**: その課金モデルが、エコシステム全体でより面白いコンテンツが生まれる循環を作るかどうか
- 先読み課金・話売り等の課金モデルは、読者の「続きが読みたい」という熱量を収益に変え、作家への還元を通じてより面白い作品の制作を促進するなら正当
- 射幸心を煽ること自体は悪ではない。それが面白さの循環に寄与するかどうかで判断する
- 会員登録なしでも作品を発見・試し読みできる導線は維持する。入口のハードルは下げ、価値を感じた読者が課金する設計

### 3.4 面白さで選ばれた作品だけを届ける

量より質を徹底する。

- 自社制作: 品質基準を満たした作品だけを公開する。月100話の生産量は目標ではなく上限
- 外部作品（Phase 2）: 最低限の品質ライン（文章の体裁・一貫性等）を設ける。基準を満たさない作品にはフィードバックを付けて差し戻す。これは検閲ではなく、読者体験の保証
- ランキング: PV数順ではなく**読了率ベース**。釣りタイトルで流入だけ稼ぐ作品が上位に来ない構造

### 3.5 作り手は問わない、面白さだけを問う

- AIで書いたか、人間が書いたか、その混合かは問わない
- 問うのは「読者が面白いと感じたかどうか」だけ
- AI利用を禁止も推奨もしない。手段は自由、結果で評価する
- このポジショニングは「AIプラットフォーム」を名乗るためではなく、**面白さの最大化に手段の制約が不要だから**

## 4. 各領域への適用

### プロダクト設計

| 判断場面 | この哲学に基づく判断 |
|---|---|
| ランキング設計 | PV順ではなく読了率順。面白さの実測値で並べる |
| レコメンド | 「人気作品」ではなく「あなたが最後まで読む確率が高い作品」 |
| トップページ | 新着順ではなく、読了率の高い作品を優先表示 |
| 検索 | ジャンル・タグに加え、「読了率○%以上」のフィルタ |
| 読書体験 | 余計なUI要素を排除。本文に集中できるリーダー画面 |

### 事業判断

| 判断場面 | この哲学に基づく判断 |
|---|---|
| KPI設定 | 最上位KPIは読了率・完読率・再訪率。MAU・PVは二次指標 |
| 外部作家の受入 | 投稿数の最大化ではなく、品質ラインの維持を優先 |
| マーケティング | 「作品数○○本」ではなく「平均読了率○○%」を訴求 |
| 収益モデル | 課金が面白さの循環（読者の対価→作家還元→より面白い作品）を生むかで判断 |
| 機能開発の優先順位 | 読了率の改善に寄与する機能を最優先 |

### 作家向けの価値提案

| 提供する価値 | 意味 |
|---|---|
| 読者行動データダッシュボード | 読了率・離脱ポイント・継続率の可視化。改善のための武器 |
| 全収益90%還元 | 業界最高の還元率。透明でシンプル |
| 品質フィードバック | 基準未達の作品には差し戻しではなく改善示唆を提供 |
| 手段の自由 | AI利用度を問わない。面白ければ正義 |

### コンテンツ制作（自社）

| 判断場面 | この哲学に基づく判断 |
|---|---|
| 生産量 | 月100話は上限であり目標ではない。品質が下がるなら減らす |
| ジャンル選定 | 「AIが書きやすい」ではなく「読者が面白がるデータがある」で選ぶ |
| 改善サイクル | 読了率の低い作品は原因を分析し、展開・テンポ・キャラを調整 |
| 公開判断 | 品質基準を満たさないエピソードは公開しない。毎日更新より品質 |

## 5. メッセージライン

### 読者向け

> **「もっと面白い小説を、すべての人に」**

面白さで選ばれた作品だけが並ぶ場所。大量の作品から探す必要はない。

### 作家向け

> **「面白さだけで評価される場所」**

フォロワー数も投稿頻度も関係ない。読者が面白いと感じたかどうか。それだけ。

### 対外的なポジショニング

「AIプラットフォーム」を名乗らない。「AI利用を問わない」は補足情報として伝える。前面に出すのは常に「面白さ」。

---

## 6. この哲学が禁じること

以下は、短期的な収益やKPI改善に寄与しても採用しない。

- **面白さの循環を断つ課金**: 収益が作家還元やコンテンツ改善に還流せず、プラットフォームだけが利益を取る構造
- **広告過多**: 読書体験を毀損する量の広告挿入
- **PV至上のランキング**: 釣りタイトルを助長するランキング設計
- **無審査の大量投稿**: 読者体験を劣化させる低品質作品の放置
- **エンゲージメントハック**: 通知の過剰送信、スクロール誘導等の暗いパターン

---

## 7. この哲学の検証方法

哲学が機能しているかどうかは、以下で検証する。

- **読了率が継続的に改善しているか**（四半期ごとにレビュー）
- **再訪率が上昇しているか**（読者が戻ってきているか）
- **読者からの定性フィードバック**（「面白い作品が見つかる」という評価）
- **作家からの定性フィードバック**（「データで作品が改善できた」という評価）

数値が改善していない場合、哲学の適用に問題がないかを振り返る。哲学自体を疑うのは最後。
